# Dev-Loop v2.1 設計規格 — 狀態機語義修復 + loop-native 重構

- **日期**:2026-07-02
- **基線**:[2026-06-30-dev-loop-v2-design.md](2026-06-30-dev-loop-v2-design.md)
- **產物形態**:設計文件 / 規格(spec);本份不含實作
- **切分**:兩個 OpenSpec change,依序跑兩輪 dev-loop
  - **Change A**:狀態機語義修復(engine-semantics-fixes)
  - **Change B**:loop-native 重構(auto-arm + SKILL loop 敘事)— 依賴 A 先 merge

## 1. 動機

v2 引擎驗收後發現三個確定性缺口與一個編排層脆弱點:

1. **propose 死路(已重現)**:`statemachine.py` 沒有任何從 `propose` 出發的 event。blocking(proposal) 把 phase 推到 `propose` 後,重跑 `proposal-review` 得到 `error: no transition from 'propose' on 'propose_clean'`(exit 2)。SKILL 宣稱的「自動重新 propose 後再 proposal-review(計數,上限 N)」引擎層不可達,且重試計數未實作。
2. **gate↔fix 內圈不受迴圈保護**:`iteration` 只在 `gate_pass` 時 +1。測試永遠紅時 `gate_fail → fix → fix_done → gate → gate_fail → …` 無限迴圈,永不升級。
3. **escalated 是死胡同**:無 outbound 轉移;人工介入處理後只能重跑 `start` 整個重來。
4. **手動 arm 儀式是最大脆弱點**:SKILL 要求模型在 ~10 個寫 checkpoint 的命令後記得手動 arm;漏 arm → 續跑不啟動,靜默失敗。違反「確定性歸引擎」哲學。另 `trigger=harness` 路徑(每 checkpoint 手動 ScheduleWakeup)在手工重造 harness 原生的 /loop dynamic mode。

另收斂 memory 中已知小項:`finish` 值域無 validation(typo 靜默退化成 ask-like)、`render_followup` 無 trailing newline、cli.py import 順序。

## 2. Change A — 狀態機語義修復

### 2.1 統一 pattern

**計數存 checkpoint、判斷在 cli 層、transition 純函式表只加離散 event。**
`transition(phase, iteration, event, max_iterations)` 簽名不變;cli 依 checkpoint 計數決定送哪個 event。

### 2.2 checkpoint 新欄位

```jsonc
"propose_attempts": 0,   // proposal_review blocking(proposal) 回 propose 的累計次數
"gate_failures": 0       // gate_fail 累計次數(change 生命週期內,不隨 gate_pass 重置)
```

缺欄位視為 0(v2 checkpoint 向後相容)。

### 2.3 transition 表新增

| 現 phase | event | 新 phase | 說明 |
|---|---|---|---|
| `propose` | `propose_done` | `proposal_review` | 補回路:重新 propose 完成後由 SKILL 發此 event 再跑 proposal-review |
| `proposal_review` | `propose_retry_exceeded` | `escalated` | cli 在 blocking(proposal) 且 `propose_attempts` 超上限時改發此 event |
| `gate` | `gate_retry_exceeded` | `escalated` | cli 在 gate 失敗且 `gate_failures` 超上限時改發此 event |
| `escalated` | `human_resume_propose` | `propose` | 人工介入後續跑(提案層問題) |
| `escalated` | `human_resume_fix` | `fix` | 人工介入後續跑(實作層問題) |

- brainstorm 層級的重啟維持人工重跑 `start`(沿 v2 註記,brainstorm 無 inbound auto-transition)。
- `human_resume_*` 套用時,cli 把 `iteration` / `propose_attempts` / `gate_failures` 三個計數歸零(否則 resume 後立即再超限)。

### 2.4 cli 行為

- `_cmd_proposal_review`:classify 得 `propose_blocking_proposal` 時 `propose_attempts += 1`;若超上限(`--max-propose`,預設 3)改 apply `propose_retry_exceeded`。
- `_cmd_gate`:失敗時 `gate_failures += 1`;若超上限(`--max-gate`,預設 3)改 apply `gate_retry_exceeded`(仍印失敗輸出、exit 1)。
- `_cmd_event`:接受新 events;`human_resume_*` 時歸零三計數。

### 2.5 finish validation + 小項

- `resolve_finish`:值不在 `{merge, pr, ask, None}` → 拋明確錯誤;cli `finish` 印 `error: invalid finish value <v>` 到 stderr、exit 2。不再靜默退化。
- `render_followup` 補 trailing newline;cli.py import 順序整理。

### 2.6 SKILL 對應(隨 Change A 一併更新源頭 skills/dev-loop/SKILL.md)

- 步驟 4 blocking(proposal) 路徑補:「重新 propose 完成後 `event --event propose_done`,再跑 proposal-review」。
- 升級敘述補 `human_resume_propose` / `human_resume_fix` 的人工續跑出口。

## 3. Change B — loop-native 重構

### 3.1 引擎自動 arm(兜底層)

- 抽出 `ensure_armed(checkpoint_path, heartbeat)`(現 `_cmd_arm_local` 主體,idempotent:watcher 存活即 no-op)。
- **所有寫 checkpoint 的子命令**(`start`/`event`/`gate`/`proposal-review`/`qa`/`leg-done`/`review`/`units-*`/`unit-*`)在 save 後、`resume_exec` 非空且 config `auto_arm` 為 true(預設)時自動呼叫。
- arm 失敗僅 stderr 警告,不影響主命令 exit code(arm 是兜底,不能反噬主流程)。
- config 新增 `auto_arm`(預設 true);`arm-local` 子命令保留(手動補救用)。

### 3.2 SKILL 重構成 loop 敘事

核心迴圈改寫為:

> **每回合:讀 checkpoint phase → 推進到下一個卡點(✋人工關卡、`done`/`escalated` 終態、或 token 邊界)→ 未到終態則 ScheduleWakeup 排下一回合(長 fallback ≥1200s)。**

- `/loop /dev-loop resume` 成為標準運行形態;✋關卡與終態不再排程,loop 自然收束(與 /loop「omit 即結束」語義契合)。
- **刪除「每個 checkpoint 後 arm」整節**(引擎已自動)。
- 「Token 用罄續跑」節縮寫為:watcher 兜底(引擎自動)+ /loop 正職(harness 排程)。

### 3.3 resume 決策下沉引擎

`status` 子命令擴充輸出 `next:` hint——引擎依 phase(與 units/legs 狀態)印出建議的下一步命令,例:

```
phase=gate iteration=2
next: python3 -m devloop.cli gate --file .devloop/checkpoint.json --cmd "<test-cmd>" …
```

冷啟動 resume 讀 hint 行動;SKILL 的 resume 節縮成「跑 `status`,照 `next:` 做」。判斷型步驟(dispatch subagent 的內容)仍由 SKILL 描述,hint 只給命令骨架。

### 3.4 trigger 語義(向後相容)

| trigger | 語義 |
|---|---|
| `local`(預設) | watcher only(headless / `claude -p` 環境) |
| `harness` | watcher 兜底(auto_arm 照常)+ /loop 正職:交互 session 以 `/loop /dev-loop resume` 運行 |

## 4. 測試策略

stdlib pytest 全程 TDD(dogfooding):

- transition 新轉移表逐條單測(含 InvalidTransition 邊界)。
- 計數:propose_attempts / gate_failures 累加、超限改發 exceeded event、`human_resume_*` 歸零。
- checkpoint 新欄位序列化 + 缺欄位向後相容(視為 0)。
- auto-arm:注入 fake spawn/pid-alive,驗每個寫 checkpoint 的子命令 save 後呼叫 `ensure_armed`;arm 失敗不改主 exit code;`auto_arm=false` 不呼叫。
- finish validation:invalid 值 exit 2 + stderr;合法值不受影響。
- `status` next hint:各 phase 輸出契約。
- SKILL.md 屬文檔改動,由 review leg 把關一致性。

## 5. 各步驟模型選擇

| dev-loop 步驟 | 模型 | 理由 |
|---|---|---|
| 編排 / Brainstorm / Propose | Fable(主編排) | 判斷最重;高於 skill 預設 Opus,免 dispatch |
| Proposal Review | Opus subagent(冷啟動) | 獨立視角把關 |
| Apply(TDD) | Sonnet subagents | 機械性 red→green |
| QA Gate | Sonnet subagent | 執行驗證型 |
| Code Review leg | Opus subagent(冷啟動) | 判斷型 |
| Fix | 機械 Sonnet / 架構 Fable | 依 blocking 性質 |
| uiux leg | 不啟用 | 純引擎 + 文檔,needs_uiux=false |

## 6. 向後相容

- v2 checkpoint 缺新計數欄位 → 視為 0。
- `auto_arm` 未設 → true;`trigger` 未設 → local。
- 既有 events / 轉移全部保留;新 events 只擴充表。
- `finish` 合法值行為不變;僅 invalid 值從靜默退化改為報錯。
